# Upstream 1.0.79 feature recovery execution plan

This plan implements the
[upstream 1.0.79 recovery specification](../../product-specs/upstream-1.0.79-recovery.md)
as test-driven semantic ports. Implementation starts only after the user
approves this plan and its proposed scope.

## Goal

Recover all applicable functionality from the supplied Claude in Chrome 1.0.79
package without regressing Claw in Chrome's current behavior, readable recovery
architecture, security controls, or release packaging.

## Constraints

Implementation must preserve the current source of truth and make every source
decision reviewable.

- Treat `C:\Users\leo\Downloads\Claude-Chrome` as read-only source evidence.
- Don't copy the source directory, replace all generated bundles, or overwrite
  the current manifest and page shells.
- Preserve every unrelated working-tree change and all current local features.
- Write one focused failing behavior test before each implementation slice.
- Prefer readable modules and use generated bundle edits only when no stable
  extension seam exists.
- Add a semantic anchor test and recovery-model note for every bundle patch.
- Treat permissions, OAuth credentials, managed policy, content security
  policy, and remote connections as security-sensitive changes.
- Keep each slice independently revertible and commit only coherent GREEN
  checkpoints.
- Run `npm run validate:fast` before each implementation checkpoint and
  `npm run validate:full` for the final runtime-sensitive result.

## Execution slices

Each implementation slice begins with one RED behavior and ends with focused
GREEN evidence.

### Slice 1: Build an executable behavior matrix

This slice turns the opaque package diff into an auditable set of decisions.

1. Inventory source and current manifest fields, page entry points, loaded
   assets, runtime message names, storage keys, policy names, and public UI
   behaviors.
2. Classify each functional delta as missing, locally equivalent, divergent,
   or excluded, with an owning module and test target.
3. Add a regression test that fails when an accepted source entry point or
   dependency has no mapped owner or release-package path.
4. Record non-functional hash churn separately so it can't justify broad
   generated-file replacement.

### Slice 2: Recover managed enterprise policies

This slice adds blocked-URL and forced-organization controls without changing
normal unmanaged behavior.

1. Test URL wildcard normalization, case handling, malformed values, live
   managed-storage updates, and unaffected URLs.
2. Test allowed, rejected, missing, malformed, and multiple organization UUID
   policies through the public login boundary.
3. Implement readable policy helpers and the narrow runtime integration.
4. Add the `storage.managed_schema` manifest declaration and document policy
   behavior without adding unrelated permissions.

### Slice 3: Recover MCP OAuth through Chrome identity

This slice ports the source PKCE flow through the current MCP and permission
boundaries.

1. Test verifier and challenge generation, state validation, redirect parsing,
   cancellation, token exchange, refresh, and logout behavior.
2. Test that every sensitive field is redacted from console and persisted
   diagnostics, including error paths.
3. Add `identity` only with the accepted OAuth path and update the intentional
   manifest baseline.
4. Verify existing custom-provider credentials and native-host MCP behavior are
   unchanged.

### Slice 4: Recover the Claude.ai onboarding bridge

This slice ports the source content-script loader and validated starter tasks.

1. Test known and unknown task identifiers at the content-script boundary.
2. Test tab selection, side-panel opening, prompt delivery retries, and
   service-worker restart behavior.
3. Implement the smallest readable loader, prompt map, and background message
   handler compatible with the current runtime contract.
4. Update web-accessible resources and release inventory only for required
   files.

### Slice 5: Recover accepted artifact rendering additions

This slice adds Mermaid and any other matrix-approved artifact behavior without
replacing the current visualizer or side-panel recovery.

1. Test accepted MIME types, sanitization, CSP isolation, diagram limits,
   malformed content, theme changes, and render cancellation.
2. Test lazy chunk and stylesheet dependencies through the packaged extension,
   not only the checkout tree.
3. Implement readable rendering adapters where possible and use narrow bundle
   anchors only for generated React ownership.
4. Verify existing GIF, tool-result, Markdown, and interactive artifact paths
   remain functional.

### Slice 6: Decide opaque and cross-cutting source deltas

This slice prevents filenames and minified symbols from becoming requirements
without understood behavior.

1. Establish the public behavior, data flow, endpoints, and security impact of
   `Conway`, `squares`, connector UI changes, and remaining matrix entries.
2. Present any material scope expansion for user approval before implementing
   it.
3. Port accepted items as additional test-first vertical slices.
4. Record excluded or already-equivalent items with evidence and user impact.

### Slice 7: Integrate, inspect, and review

This slice proves the combined extension remains safe, loadable, and
recoverable.

1. Update `docs/recovery-model.md`, `docs/SECURITY.md`,
   `docs/RELIABILITY.md`, and `docs/QUALITY_SCORE.md` for final behavior and
   remaining debt.
2. Run focused regression suites, then `npm run validate:fast`.
3. Run `npm run validate:full` and exercise enterprise policy, OAuth,
   onboarding, artifact, options, and side-panel flows in the unpacked release
   package.
4. Run `npm run inspect:runtime` and compare permissions, policies, entry
   points, and diagnostic surfaces against the approved matrix.
5. Perform spec, harness, architecture, security, reliability, and regression
   review passes, then rerun affected validation.
6. Move this plan to `docs/exec-plans/completed/` with RED, GREEN, runtime,
   review, commit, and push evidence.

## Harness checklist

- [x] The feature brief records the source baseline, goal, non-goals, user
  contract, risks, and rollback before implementation.

  and done criteria before implementation.
- [x] An executable source-to-local behavior matrix accounts for every
  functional delta and explicit exclusion.
- [x] Managed-policy tests cover matching, updates, malformed data, and normal
  unmanaged behavior.

  validation.
- [ ] Every generated bundle patch has a semantic anchor regression test.
- [ ] `npm run inspect:runtime` reports managed policy, permission, entry-point,
  and diagnostic changes clearly.
- [ ] Headed E2E covers accepted UI and extension-loading behavior from a

  policy, and failure-state changes.

provide deterministic evidence.
